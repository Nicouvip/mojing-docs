# 墨境 Agent 总管降级协议 v1.0

> **制定者**：墨境架构审查员 + 协议执行官  
> **最后更新**：2026-06-26  
> **状态**：✅ 已批准，即时生效

---

## 三句核心规则

1. **总管离线超过30分钟、连续3次调度错误、或用户主动触发 → 立即降级，不得等待。**  
2. **降级后架构审查员自动升任临时总管——只读现存任务文件、维持集群不崩，但不派新活、不改任务文件。**  
3. **总管回来第一件事是读降级日志，判断是恢复原调度还是推倒重排；日志不到位，不准恢复总管权限。**

---

## 一、触发条件（任一满足即降级）

| 条件 | 判定方式 | 阈值 |
|:-----|:---------|:----:|
| 🕐 **总管离线** | 集群监控轮询 `总管心跳` 超时无响应 | 连续 **> 30 分钟** |
| 🔁 **连续调度错误** | `dispatch.py` 返回错误码，或分配的任务连续出Bug | **连续 3 次** |
| 🫵 **用户主动触发** | 用户在任意窗口输入 `触发降级` 或 `总管接管` | 即时生效 |
| ⚠️ **总管自检挂掉** | 总管自检清单连续 2 轮未通过（见集群治理协议第7章） | 2 轮未通过 |

**检测者**：协议执行官（每 5 分钟轮询一次集群健康状态）  
**确认后操作**：协议执行官写入 `output/降级日志.md` → 通知架构审查员接任

## 二、降级后谁接管

### 2.1 接班人：架构审查员 → 临时总管

| 项目 | 内容 |
|:-----|:------|
| **接管者** | 🖥️ 架构审查员（Architecture Reviewer） |
| **晋升条件** | 无需额外审批，协议执行官确认触发条件后即生效 |
| **任期** | 直至总管恢复或用户指定新总管 |
| **权力来源** | 本协议 v1.0 第 2 章 |

### 2.2 临时总管权限矩阵

| 权限 | 允许 | 禁止 |
|:-----|:----:|:----:|
| 读取 `tasks/` 下所有角色任务文件 | ✅ | — |
| 读取 `output/` 最近产出 | ✅ | — |
| 读取 `context/` 集群上下文件 | ✅ | — |
| 读取 `decisions.jsonl` 决策日志 | ✅ | — |
| 运行 `dispatch.py progress` 查看集群状态 | ✅ | — |
| 修改 `tasks/角色名.md` | ❌ | 任务文件是总管的调度遗产，只读不改 |
| 派发新任务 | ❌ | 这是总管的专属权力 |
| 修改代码 | ❌ | 越界，见集群治理协议第9章 |
| 修改本协议 | ❌ | 协议变更需总管 + 用户双签 |
| 标记角色为「暂停」或「等待」 | ✅ | 写入 `context/broadcast.json` 告知全集群 |

### 2.3 临时总管职责

1. **稳定集群**：确认所有正在执行的角色不会因总管缺席而卡死
2. **只读盘点**：读一遍所有 `tasks/角色名.md` 和 `output/` 最新产出，搞清楚总管走之前留了什么摊子
3. **广播降级**：写入 `context/broadcast.json` → `{ "event": "steward_degraded", "temp_steward": "架构审查员", "timestamp": "..." }`
4. **等待**：不派新活、不改任务文件、不催角色，除非角色主动求助
5. **紧急刹车**：如果某个角色出现严重越界（正在删库、改生产配置），临时总管有权喊停

## 三、降级期间各角色如何处理

### 3.1 窗口角色（UI设计师 / UX体验员 / QA测试工程师 / 协议执行官）

```
状态：继续等
行为：保持窗口打开，不做新任务
例外：已在进行中的走查/测试，可以完成当前轮次并产出
      完成后自动进入等待，不主动索要新任务
```

### 3.2 Task 小弟（后端技术负责人 / 前端技术负责人 / 提示词系统专家 / 产品经理）

```
状态：暂停新任务
行为：收到 `broadcast` 降级通知后：
      1. 如果当前正在执行任务 → 完成当前步骤，不开始下一步
      2. 如果空闲 → 回复「已暂停，等待总管恢复」
      3. 不主动发起新任务、不修改任务文件、不跨角色联动
例外：如果当前任务只差最后一步（如 pnpm build 验证），允许跑完再停
```

### 3.3 临时总管（架构审查员）

```
状态：监护模式
行为：每 15 分钟轮询一次总管是否恢复
      记录降级期间集群发生的任何异常到 `output/降级日志.md`
      不派活、不改任务、不替角色做决策
```

### 3.4 总管（离线中）

```
状态：不可用
行为：恢复后主动读取降级日志，走恢复流程
```

## 四、恢复流程

### 4.1 总管恢复判定

| 检测方式 | 标准 |
|:---------|:-----|
| 总管主动报到 | 总管在任意窗口回复「我回来了」 |
| 集群监控感知 | `总管心跳` 恢复响应 ≥ 3 次连续正常 |
| 用户确认 | 用户告知「总管已上线」 |

### 4.2 恢复步骤

```
第1步：总管读取 output/降级日志.md
      └─ 问：「我离线期间发生了什么？谁做了什么？」
第2步：临时总管（架构审查员）口头汇报集群状态
      └─ 哪些角色已完成、哪些卡住、哪些处于暂停
第3步：总管判断恢复策略
      ├─ 轻量恢复：降级期间无异常 → 继续原调度
      └─ 完整重排：降级期间有任务中断/角色卡死 → 重新调度
第4步：总管撤销广播
      └─ 写入 context/broadcast.json → { "event": "steward_restored" }
第5步：临时总管交还权限
      └─ 架构审查员恢复原角色，不再行使总管权力
第6步：总管处理积压
      └─ 检查 dispatch.py progress
      └─ 重新分配中断的任务
      └─ 更新决策日志
```

### 4.3 恢复失败兜底

如果总管恢复后仍连续出错（再次触发降级条件）：

| 情况 | 操作 |
|:-----|:-----|
| 同一问题重复 2 次 | 用户介入，指定新总管 |
| 总管主动交出权力 | 用户指定新总管，或架构审查员长期接任 |
| 用户不在场 | 架构审查员保持监护模式，每轮记录日志，直至用户上线 |

## 五、降级日志格式

每次降级事件必须记录到 `output/降级日志.md`，追加写入：

```markdown
## 降级事件 #[编号]

- **触发时间**：2026-06-26 14:30
- **触发条件**：总管离线 > 30 分钟 / 连续 3 次调度错误 / 用户主动触发
- **检测者**：协议执行官
- **临时总管**：架构审查员
- **降级时集群状态**：
  - 正在执行的角色：[角色名] — [任务]
  - 等待中的角色：[角色名]
  - 已完成的产出：[文件列表]
- **降级期间事件**：
  - [时间] [事件描述]
- **恢复时间**：2026-06-26 15:10
- **恢复方式**：总管主动报到 / 集群监控感知 / 用户确认
- **恢复策略**：轻量恢复 / 完整重排
- **备注**：
```

## 六、协议守护

| 守护规则 | 说明 |
|:---------|:-----|
| **本协议不可被临时总管修改** | 只有总管 + 用户双签才能改 |
| **降级日志永久保留** | 每次降级事件写入 `output/降级日志.md`，不删除、不覆盖 |
| **架构审查员不得拒绝接任** | 触发条件满足时，必须接任。拒绝 = 集群失守 |
| **协议执行官每轮降级后审查合规** | 检查降级流程是否按协议执行，有无越权行为 |
