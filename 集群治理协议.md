## 二、任务分配铁律

### 2.1 谁能给谁派活
- **总管** → 所有角色（唯一调度中心）
- **task 小弟之间** → 通过 `context/inbox/` 发建议，不得直接改对方任务文件
- **窗口角色之间** → 通过产出文件互相引用，不得代派
- **任何角色** → 不得给总管派活

### 2.2 派活流程
1. 总管更新 `tasks/角色名.md`（强制技能 + 具体指令 + 验证标准）
2. 🔧 task 模式：总管直接调 `task()` 子Agent，注入 `context/角色名.md` 记忆
3. 🖥️ 窗口模式：用户在对话窗口发送 `查收任务`
4. 完成后写产出到 `output/`，更新 `context/角色名.md`

### 2.3 任务文件格式
```markdown
【强制技能 — 技能1 + 技能2】
────────────────────────
T编号：任务名称

具体指令（3行内说清楚）
验证标准：怎么算完成
输出文件：output/角色-任务名.md
```

## 三、通信协议

### 3.1 消息类型
| 类型 | 含义 | 谁发 |
|------|------|------|
| `task_assigned` | 新任务已分配 | 总管 |
| `bug_report` | 发现 Bug | 任意 |
| `code_review` | 代码审查建议 | 架构 |
| `qa_request` | 请求 QA 测试 | 后端/前端 |
| `suggestion` | 改进建议 | 任意 |
| `handoff` | 交接（我做完你了接着做） | 任意 |
| `retry` | 自动重试通知 | 集群监控 |
| `broadcast` | 全集群通知 | 总管 |

### 3.2 消息路由
- **task → task**：写入 `context/inbox/目标角色.jsonl`
- **task → 窗口**：写入 `context/团队/suggestions.md`，下一轮窗口会话自动注入
- **窗口 → task**：窗口产出文件被总管读取后注入 task prompt
- **广播**：`context/broadcast.json` → 全集群可见

### 3.3 信息流转
```
总管分配 → task 小弟改代码 → QA 自动门禁 → 架构终审 → 总管合并备份
                ↕ (via context/)
          窗口角色检测 → UX 走查 → UI 审查 → 反馈回 task
```

## 四、质量门禁

### 4.1 Task 小弟完成前自检（必须）
1. `pnpm build` 零错误（或 `tsc --noEmit`）
2. 改动范围不超过任务范围（不要顺手改无关文件）
3. 产出文件写入 `output/`
4. 上下文记忆写入 `context/角色名.md`
5. 如果改动涉及已有功能 → 自动 `trigger_qa("角色", "改了什么")`

### 4.2 窗口角色完成前自检
1. 强制技能包已调用（任务文件头部标注的）
2. 产出文件去日期、四要素齐全
3. gstack 截图（UX/UI）
4. 更新 Bug 清单（QA）

### 4.3 总管验收标准
- 🔧 task 产出：看 `output/` 文件 + 跑 `pnpm build`
- 🖥️ 窗口产出：读产出报告 + 看截图
- 代码变更 > 50 行 → 必须走 QA + 架构双审

## 五、备份策略

### 5.1 自动备份
- **每轮总管调度后**：`git add -A && git commit`
- **每天收工前**：全量备份到 `D:/建网站/backups/`
- **重大变更前**：先 `git stash` 或打 tag

### 5.2 备份内容
| 内容 | 方式 | 频率 |
|------|------|------|
| 代码 (mojing-app) | Git | 每轮 |
| 文档+产出 (mojing-docs) | 拷贝到 backups/ | 每天 |
| 集群上下文 (context/) | Git（随 mojing-docs） | 每轮 |
| 数据库 | 暂无（localStorage） | — |

### 5.3 回滚流程
1. `git log --oneline -10` 找目标版本
2. `git reset --soft <hash>` 回滚
3. 通知受影响角色重做

## 六、故障处理

### 6.1 Task 小弟超时
- 监控线程每 30 秒轮询
- 超时 15 分钟 → 自动重试（最多 3 次）
- 3 次仍失败 → 记入决策日志 + 通知总管手动介入
- 总管可降级到窗口模式执行

### 6.2 窗口角色无响应
- 用户手动在窗口输入 `查收任务` 唤醒
- 超过 1 天未响应 → 总管在集群仪表盘标记 ⚠️

### 6.3 代码冲突
- 同一文件多人改 → 总管合并前 `git diff` 检查
- 冲突标记 `<<<<<<<` → 总管手动解决后提交

### 6.4 仪表盘挂了
- 重启：`python D:/建网站/mojing-docs/项目全景进度.py`
- 检查端口：`netstat -ano | findstr :8899`

## 八、效率规则

### 8.1 并行优先
- 🔧 task 小弟能用 `parallel_tasks` 同时跑的绝不串行
- 无依赖的窗口任务可同时发

### 8.2 上下文最小化
- task prompt 限制 5000 字以内
- 记忆注入只取最近 2000 字
- 窗口对话每 500 条提醒用户清理

### 8.3 避免重复
- 任务分配前先 `dispatch.py progress` 检查谁空闲
- 同一文件 2 人不能同时改
- QA 和架构审查错开 5 分钟以上

## 九、禁止事项

| 禁止 | 原因 |
|------|------|
| ❌ 总管不动手改代码 | 越界（除非用户明确要求） |
| ❌ task 小弟改 3 个以上无关文件 | 扩散风险 |

## 十四、协议执行官

### 14.1 定位
> 🖥️ 独立窗口，不受总管调度。职责：监督全集群合规。直接向总管和用户汇报。

### 14.2 与总管区别
| | 总管 | 协议执行官 |
|------|------|------|
| 职责 | 调度任务、推进项目 | 监督合规、纠正违规 |
| 看什么 | 产出质量、进度 | 边界、通信、门禁 |
| 权力 | 分配任务 | 提出警告、要求整改 |

### 14.3 监督三项
| 监督项 | 协议条款 | 检测方式 |
|------|------|------|
| 职责越界 | 第9章 | `git diff --stat` 查文件范围 |
| 通信违规 | 第3章 | 轮询 inbox/outbox 写入记录 |
| 门禁跳票 | 第4章 | 检查 output/ 时间戳 + build 日志 |

### 14.4 自约束（谁监督监督者）
1. 产出全透明：合规报告写入 `output/`，全集群可读
2. 架构互审：架构审查员定期检查协议执行官是否越界
3. 双终结权：总管验收 + 用户可直接否决。纠正2次→协议升级

### 14.5 触发方式
用户在该角色窗口输入 `查收任务`。每周至少产出 1 份合规报告。

## 十一、多路审议制度（铁律）

**核心原则**：任何重要产出必须经多角度碰撞，不得独断。包括总管自己。

### 升级：遇到难题 → 开会，不是硬扛
- **任何角色卡住超过 2 轮** → 触发会议室
- **总管做重大决策前** → 必须至少 3 人出席会议讨论
- **用户提出新方向** → 不能直接执行，先开会讨论方案
- **出现 Bug 修了 2 次没好** → 开会找根因

### 讨论质量铁律（新增）
- **最低 4 轮**：不够 4 轮不准关闭会议室，总管也不行
- **关闭条件**：连续 2 轮无新观点出现 → 才能总结关闭
- **用户有最终否决权**：用户可以随时推翻讨论结果、注入新观点、要求重开会议
- **参会人数**：不少于议题相关角色的一半 + 1
- **每轮必须换角度**：上一轮挑毛病的，下一轮必须出方案；上一轮出方案的，下一轮必须挑毛病

### 11.1 什么时候触发多路审议
| 场景 | 触发条件 | 审议人数 |
|------|------|:---:|
| 新功能设计 | 任何新功能 | 4人（激进/保守/UX/挑刺） |
| 架构决策 | 涉及 3+ 文件或新依赖 | 3人 |
| Bug 修复方案 | 高危 Bug（P0） | 3人 |
| 总管调度决策 | 涉及 3+ 角色重新分配 | 3人（其他角色评审总管） |
| 产品方向 | 新增页面/功能模块 | 3人 |
| 每周回顾 | 每周日晚 | 全员8人 |

### 11.2 审议流程
```
需求确认 → 总管分配角度
  ├─ A（激进/最优方案）
  ├─ B（保守/最小改动）
  ├─ C（UX/用户视角）
  └─ D（挑刺/风险扫描）
         ↓
  各产出方案（并行，max_steps=5）
         ↓
  总管合成 → 输出最佳方案
         ↓
  拆分实现 → 多人并行执行
         ↓
  双重验收（QA + 架构）
```

### 11.3 审议输出格式
每次审议产出必须包含：
- 各方方案摘要（≤3句话）
- 共同点和分歧点
- 最终选择 + 理由
- 被否决方案的保留价值（什么时候可能重新考虑）

## 十二、互相监督制度

### 12.1 总管受监督
- **每轮调度后**：架构审查员检查调度合理性（有无冲突、有无遗漏）
- **每日收工前**：产品经理对照 Sprint 计划检查是否有任务偏离
- **重大决策后**：QA 测试工程师验证决策是否引入新风险
- **连续 3 次同样错误**：任何角色可向用户建议暂停该方向

### 12.2 角色间互相监督
- 前端产出 → 后端验证 API 调用是否合理
- 后端产出 → 前端验证接口是否可用
- 任何角色发现另一角色越界 → 发 `context/inbox/角色名.jsonl` 提醒
- UI 和 UX 互相审核视觉和交互一致性

### 12.3 自动化监督
- 集群监控线程每 30s 轮询健康
- QA 请求 15min 未响应 → 自动升级到总管
- 代码变更 > 100 行 → 强制触发架构审查
- 同一文件 2 人同时修改 → 总管告警

## 十三、提醒与纠错

### 13.1 提醒清单（每次调度前总管必查）
```
□ 有没有忘了的任务？（查 dispatch.py progress）
□ 有没有冲突的文件？（查 git status）
□ 有没有逾期未响应的？（check_health > 15min）
□ 有没有连续失败 2 次以上的？（check _retry_count）
□ 上次总管自检完成了没有？
```

### 13.2 纠错流程
1. 发现错误 → 立即记录到 `decisions.jsonl`
2. 通知受影响角色（通过 inbox）
3. 回滚受影响代码（git reset）
4. 重新分配 + 记录纠错原因
5. 更新协议防止同类错误

### 13.3 用户纠错权限
- 用户随时可以否定任何角色产出
- 用户可以直接在角色窗口输入修正指令
- 总管必须记录用户的每次纠正（为什么、怎么改）
- 同一问题被用户纠正 2 次 → 协议升级
