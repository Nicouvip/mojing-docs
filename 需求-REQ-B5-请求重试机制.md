# 需求文档：AI 请求重试机制

## 元信息
- **需求编号：** REQ-B5
- **优先级：** P1 (Should)
- **RICE 评分：** 12/100
- **涉及角色：** 后端技术负责人
- **预估工作量：** 0.5 人天
- **依赖：** 可在 A4（超时处理）完成后做，因为重试逻辑与超时逻辑共享 AbortController

---

## 用户故事

> **As a** 网络不太稳定的作者
> **I want** AI 请求失败后能自动重试一次，而不是直接报错
> **So that** 偶发的网络波动不会打断我的写作状态

**现状问题：** 当前 AI 接口的 fetch 如果失败（网络错误、500 错误），直接显示"请求失败"，用户需要手动再点击一次按钮。对于偶发的 API 抖动，自动重试一次即可解决。

---

## 功能拆解

### 5.1 前端：重试逻辑实现
文件：`src/app/editor/[id]/page.tsx`（调用 AI 接口的地方）

**自动重试策略：**
```typescript
const MAX_RETRIES = 1;  // 自动重试 1 次

async function fetchWithRetry(url: string, options: RequestInit, retries = MAX_RETRIES): Promise<Response> {
  try {
    const res = await fetch(url, options);
    if (!res.ok && retries > 0) {
      // 等待 1 秒后重试
      await new Promise(r => setTimeout(r, 1000));
      return fetchWithRetry(url, options, retries - 1);
    }
    return res;
  } catch (err) {
    if (retries > 0) {
      await new Promise(r => setTimeout(r, 1000));
      return fetchWithRetry(url, options, retries - 1);
    }
    throw err;
  }
}
```

### 5.2 重试次数与间隔
| 参数 | 值 | 说明 |
|------|:---:|------|
| 最大自动重试次数 | 1 次 | 只重试一次，避免重复请求过多 |
| 重试间隔 | 1 秒 | 给网络恢复留出时间 |
| 重试条件 | 网络错误 + HTTP 5xx | HTTP 4xx 不重试（参数错误），2xx/3xx 不重试 |

### 5.3 重试次数展示（可选）
- 如果重试成功，用户无感知，不需要任何提示
- 如果重试仍失败，显示"请求失败（已重试 1 次）"
- 让用户知道系统已经自动尝试过了

### 5.4 手动重试
在超时处理需求（A4）中已经要求了"重试"按钮，B5 与之配合：
- 自动重试 1 次失败后，显示错误信息 + "重试"按钮
- 用户点击"重试"发起手动重试（手动重试不再自动重试）

## 验收标准

1. ✅ AI 续写/润色/扩写/脑洞喷射 4 个接口，首次网络失败后自动重试 1 次
2. ✅ 重试成功后返回正常结果，用户无感知
3. ✅ 重试仍失败时，错误提示包含"已重试 1 次"信息
4. ✅ HTTP 4xx 错误不重试（用户可在前端修正输入）
5. ✅ 自动重试与超时处理（A4）共存：超时不算重试条件，重试次数不消耗在超时上

---

## 边界条件

- 幂等性：续写接口每次生成结果不同。如果第一次生成了一部分结果然后超时，重试会重新生成不同的结果。当前为非流式模式，不存在"部分结果"问题，重试安全
- 重复提交：用户手动点击"重试"时，不再自动重试（避免无限循环）
- 快速双击：用户手动快速点两次 AI 按钮 → 上一次请求未完成时应取消旧请求（参考 A4 的 AbortController 设计）
