## 元原则 · 治理的灵魂

> **治理的目的是提升效率和质量，不是为了治理而治理。**  
> 任何条款如果让团队效率下降、让成员感到束缚大于保护，就应该被质疑和修改。  
> 精简总纲不是砍条款，是回归治理的本源。

三条子原则贯穿全纲：
1. **默认公开** — 信息、代码、文档、决策默认全员可见，除非有明确的保密理由
2. **文档优先** — 任何讨论超过 15 分钟没有产出文档记录视为无效讨论；任何决策没有附带决策理由文档视为未做决策
3. **独行快，众行远** — 协作不是放弃独立思考，是让每个人的思考被更多人看见

## 一、红线章

> **原则**：红线不可触碰，触碰必有后果。总纲只写红线，操作细则由各领域责任人维护。

### 1.1 密钥与数据安全（P0）

| 红线 | 说明 | 责任人 |
|:----|:-----|:------:|
| **零密钥硬编码** | 任何密钥、Token、密码不得出现在代码仓库中；必须在 `.env` 中管理；PR 必须通过 gitleaks 等自动化扫描才能合并 | 后端技术负责人 |
| **数据隐私** | 用户数据不得用于未经用户同意的目的；数据库变更必须自动通知相关技术负责人 | 数据库工程师 |
| **三不原则** | 数据不丢失、不泄露、不滥用 | 数据库工程师 |

**违反后果**：P0 级违规，立即打回整改 + 协议执行官记违规一次。

### 1.2 技术健康度红线（P0）

| 红线 | 说明 | 责任人 |
|:----|:-----|:------:|
| **类型安全** | 所有 TypeScript 代码必须开启 `strict` 模式，禁止 `any` 类型在正式代码中出现 | 架构审查员 |
| **密钥扫描** | PR 合并前自动扫描密钥，不通过自动阻断 | CI_CD 工程师 |
| **代码评审** | 所有合并到主分支的 PR 必须至少经过一名非本人的审查员批准 | F组-审查组长 |
| **组件复用** | 组件库复用率低于 60% 的新页面不得提交（需 UI 设计师特批可豁免） | 前端技术负责人 |

**违反后果**：P0 级违规，CI/CD 流水线自动阻断合并 + 协议官记录警告。

### 1.3 质量门禁红线

| 红线 | 说明 | 责任人 |
|:----|:-----|:------:|
| **测试通过率** | 自动化测试套件通过率 ≥ 90%；P0 功能测试通过率 100% | QA 测试工程师 |
| **发布条件** | P0 级 Bug 未修复不得发布；回归测试通过率低于 95% 自动阻断发布；变更必须有回滚方案 | QA / CI_CD |
| **流水线强制执行** | 所有质量门禁（测试通过率、密钥扫描、技术债务阈值）直接写入 CI/CD pipeline，不通过自动阻断合并 | CI_CD 工程师 |

### 1.4 体验基线红线

| 红线 | 阈值 | 责任人 |
|:----|:----|:------:|
| **页面加载** | P75 ≤ 3 秒（超过 3 秒视为体验事故） | 前端技术负责人 |
| **操作路径** | 超过 5 步的功能必须走 UX 评审 | UX 体验员 |
| **交互延迟** | 编辑器核心组件渲染延迟 ≤ 16ms（60fps 标准） | 编辑器专责 |
| **动效时长** | 入场 300ms / 反馈 150ms / 转场 400ms | 动效交互设计师 |
| **状态完整性** | 关键流程（注册、写作、发布）必须有空状态 / 错误状态 / 加载状态设计 | UX 体验员 |

### 1.5 行为红线

| 红线 | 说明 |
|:----|:-----|
| **尊重沟通** | 评审代码时对事不对人，禁止人身攻击 |
| **信息透明** | 工作进度如实记录，隐瞒问题比问题本身更严重 |
| **角色边界** | 禁止读取不是自己写的输出文件；禁止在未接到指令时主动浏览全局项目状态 |
| **越界即停** | 发现自己在做不属于自己的事 → 立即停止 → 回复「越界了，请总管确认」 |

## 二、权责章

> **原则**：总纲只写权责，不写流程。谁对什么负责、谁能决定什么。

### 2.1 管理链条

```
CEO（你）
  │
总管（项目总管）—— 调度中枢，唯一调度中心
  │
副手（调度副手）—— 拆解任务，派给组长（调度的具体执行）
  │
组长（A~G 共 7 组）—— 拆派给一线
  │
一线员工（31 人）
```

**权责对应**：调度失误导致项目延期，责任人是项目总管而非执行者。权责对等——有调度权即有调度责任。

### 2.2 监督层（独立于管理链，不隶属总管调度）

```
协议执行官 🛡️     —— 监督合规、纠正违规（独立窗口直报）
组织发展官 🧑‍💼   —— 人才盘点、缺口发现、规则更新
```

### 2.3 各领域第一责任人

| 治理领域 | 第一责任人 | 责任范围 |
|:---------|:-----------|:---------|
| 调度与资源 | 项目总管 | 资源调度效率、跨组协调、品质验收 |
| 调度执行 | 调度副手 | 任务拆解、分派、进度追踪 |
| 监督合规 | 协议执行官 | 监督规则制定、违规记录、合规审计 |
| 人力与文化 | 组织发展官 | 招聘入职、能力建设、团队文化（文化为 CEO+OD 共责） |
| 架构治理 | 架构审查员 | 类型安全、技术债务总控、跨领域架构一致性 |
| 前端技术 | 前端技术负责人 | 前端代码质量、组件库规范 |
| 后端技术 | 后端技术负责人 | 后端代码质量、密钥安全、API 规范 |
| 数据库 | 数据库工程师 | 数据库变更管理、备份策略、数据安全 |
| API 治理 | API 逻辑工程师 | API 设计规范、端点测试覆盖 |
| 产品质量 | 产品经理 | 需求准入、优先级排序、功能批准（功能批准人对该功能负最终责任） |
| 质量验证 | QA 测试工程师 | 质量门禁维护、测试覆盖率（开发者对质量负责，QA 对验证负责） |
| 设计系统 | UI 设计师 | 设计系统维护、token 管理、视觉一致性审计 |
| 用户体验 | UX 体验员 | 体验基线维护、用户旅程走查 |
| 知识管理 | G组-文档组长 | 总纲维护、细则文档库、新人手册 |
| 增长度量 | 增长分析师 | 数据驱动治理、产品健康度仪表盘 |
| 运维稳定 | DevOps 工程师 | 环境管理、变更窗口、故障响应 |
| CI/CD 自动化 | CI_CD 工程师 | 流水线质量门禁强制执行、自动化工具链 |
| AI 治理 | 提示词系统专家 | 提示词资产管理、版本控制、效果评估 |

### 2.4 层级原则

| 原则 | 说明 |
|:----|:-----|
| **只派不干** | 每一层只派活、追踪、过滤问题，不亲手执行 |
| **问题过滤** | 本层能解决的不到上一层；最高层只收大事 |
| **卡住换人** | 当前层自动更换执行人，无需请示上级 |
| **能力对等** | 每层都具备派发、追踪、换人、汇报四项核心能力 |
| **单线程修改** | 同一时间只允许一个角色修改同一个文件 |

### 2.5 组长制（A~G 组）

| 组 | 组长角色 | 管理领域 |
|:--:|:---------|:---------|
| A | 前端技术负责人 | 前端开发一线 |
| B | 架构审查员 | 审查一线 |
| C | 工具组长 | 工具服务端 / API 逻辑 / 页面组件 |
| D | 基础组长 | 数据库 / DevOps / 数据迁移 |
| E | 测试组长 | QA / 自动化测试 |
| F | 审查组长 | 代码审查 / 架构审查 |
| G | 文档组长 | 文档 / 提示词 / 品牌 |

### 2.6 紧急通道

当遇到以下情况时，可绕过常规需求准入流程走紧急通道：
- **P0 级 Bug 修复**：功能完全不可用 / 数据丢失 / 安全风险
- **生产环境故障**：服务宕机、核心功能瘫痪
- **用户数据安全事件**：疑似数据泄露或未授权访问

**紧急通道流程**：
1. 发现者直接通知项目总管和产品经理
2. 总管在 15 分钟内确认是否触发紧急通道
3. 紧急任务跳过 Sprint 队列，直接分配对应角色
4. 修复后 24 小时内补录决策日志 + 需求记录
5. 每月统计紧急通道使用次数，超过 3 次/月需进行根因分析

## 三、监督章

> **原则**：监督不是为了抓错，是为了让系统健康运转。监督的最终目标是自我消亡。

### 3.1 协议执行官监督

| 维度 | 说明 |
|:-----|:-----|
| **定位** | 🖥️ 独立窗口，不受总管调度。监督全集群合规，直接向总管和用户汇报 |
| **触发** | 用户在该角色窗口输入「查收任务」；每日至少 1 份合规报告 |
| **监督三项** | 职责越界（`git diff --stat` 查文件范围）、通信违规（轮询 inbox/outbox）、门禁跳票（检查 output/ 时间戳 + build 日志） |

**职责演进路线**：
- 第 1-6 个月（严格监督期）：每日合规报告，逐项检查
- 第 7-12 个月（过渡期）：每周审计报告，重点抽查
- 第 13 个月起（审计期）：每月健康审计，以系统自检为主

### 3.2 自约束（谁监督监督者）

1. 产出全透明：合规报告写入 `output/`，全集群可读
2. 架构互审：架构审查员定期检查协议执行官是否越界
3. 双终结权：总管验收 + 用户可直接否决；纠正 2 次 → 协议升级

### 3.3 统一监督三准则

1. **违规分级**：所有违规按严重程度分 P0-P3 四级，定义和处罚标准全集团统一
2. **24 小时通知**：任何处罚决定必须在 24 小时内书面通知当事方
3. **申诉权**：被处罚人有申诉权，申诉由 CEO 终裁

### 3.4 互相监督制度

**总管受监督**：
- 每轮调度后：架构审查员检查调度合理性
- 每日收工前：产品经理对照 Sprint 计划检查任务偏离
- 连续 3 次同样错误：任何角色可向用户建议暂停该方向

**角色间互相监督**：
- 前端产出 → 后端验证 API 调用是否合理
- 后端产出 → 前端验证接口是否可用
- 任何角色发现另一角色越界 → 发 `context/inbox/角色名.jsonl` 提醒
- UI 和 UX 互相审核视觉和交互一致性

**自动化监督**：
- 集群监控线程每 30s 轮询健康
- QA 请求 15min 未响应 → 自动升级到总管
- 代码变更 > 100 行 → 强制触发架构审查
- 同一文件 2 人同时修改 → 总管告警

### 3.5 三级仲裁机制

当不同责任人之间、不同准则之间发生冲突时：

| 级别 | 方式 | 时限 | 说明 |
|:----:|:-----|:----:|:-----|
| **第一级** | 争议双方直接协商 | 48 小时 | 鼓励用 A/B 测试数据代替争论；可约定「仲裁性实验」——以最小验证实验获取决策依据 |
| **第二级** | 项目总管主持调解 | 24 小时 | 项目总管不得是争议方；如涉及调度权，调解人换为组织发展官或架构审查员 |
| **第三级** | CEO 最终裁决 | 立即 | 不可上诉；裁决基于数据而非意见 |

**仲裁原则**：
- 技术争议必须依据性能测试报告、用户数据或行业标准，而非个人意见
- 仲裁记录（脱敏后）全员可见，作为团队学习材料
- 仲裁决定具有约束力，拒不执行按 P1 级处罚

### 3.6 惩罚机制

**违规分级**：

| 级别 | 标签 | 定义 | 响应时间 |
|:----:|:----:|:-----|:--------:|
| P0 | 🔴 | 功能完全不可用 / 数据丢失 / 安全风险 | 立即 |
| P1 | 🟡 | 功能部分失效 / 体验严重受损 | 当前 Sprint |
| P2 | 🟢 | 体验问题 / 非核心功能异常 | 下个 Sprint |
| P3 | ⚪ | 视觉 / 文案 / 优化类 | backlog |

**五级递进惩罚链**：
⚠️ 警告 → 🔄 打回 → ❄️ 冻结 → ⬇️ 降级 → 🔼 协议升级

**冻结机制**：同一 task 角色连续 2 次违规（超时无响应），自动写入 `context/frozen.json`，跳过监控，总管不派新活。

**总管降级协议**：
1. 总管离线 > 30 分钟 / 连续 3 次调度错误 / 用户主动触发 → 立即降级
2. 降级后架构审查员自动升任临时总管（监护模式，只读不写不改不派）
3. 总管回来后读取降级日志 → 判断恢复策略 → 走恢复流程

### 3.7 宽限期制度

新规则执行的前 14 天为宽限期——以警告为主不记罚分，给大家适应时间。宽限期结束后从严执行。

## 四、成长章

> **原则**：治理不是管住人，是让人成长。好东西要能持续。

### 4.1 信任等级体系

基于协作积分和任务完成质量，角色可晋升信任等级：

| 等级 | 名称 | 条件 | 权限 |
|:----:|:-----|:-----|:-----|
| L1 | 新人 | 入职 < 30 天 | 需导师指导；不接触 P0 任务；前 5 个 PR 须完整审查 |
| L2 | 执行者 | 通过新人期 | 独立承担 P2 级任务；可参与代码审查 |
| L3 | 骨干 | 连续 2 个月无违规 + 协作积分 ≥ 50 | 独立承担 P1 级任务；可带新人 |
| L4 | 专家 | 季度绩效 top 20% + 有跨组贡献 | 参与架构决策；有特批豁免权 |
| L5 | 架构师 | CEO + 架构审查员双签 | 全局技术决策权；可修改技术规范 |

**协作积分**：跨组协作出力获得积分，可兑换「优先调度权」。

### 4.2 新人入职标准流程

| 时间 | 内容 |
|:----:|:-----|
| 第 1 天 | 环境搭建 + 团队介绍 + 新人手册导读 |
| 第 2-3 天 | 产品演示 + 文档体系导读 + 文档意识培训 |
| 第 1 周 | 完成第一个小任务（P2 级），熟悉开发流程 |
| 第 2 周 | 开始独立承担 P2 级任务，配资深搭档 |

- 新人标记：调度系统中标记为「新人」，自动匹配同组资深搭档
- 新人前 5 个 PR 须经组长亲自完整审查（审查时附带修改原因解释）

### 4.3 知识管理制度

1. **知识资产归档**：所有设计决策、架构方案、技术选型理由、业务流程必须文档化存储
2. **AB 角制度**：每个关键模块至少有两人了解
3. **内部分享**：核心知识持有者每双周做 30 分钟分享，分享内容必须产出文档
4. **提示词资产管理**：所有提示词集中在统一目录按版本管理，每次修改记录变更原因和预期效果
5. **文档优先**：知识的提供者负责知识的准确性和时效性，知识的消费者负责反馈知识的缺陷

### 4.4 度量体系（质量信号灯）

采用「绿 / 黄 / 红」三档信号灯，分三阶段推进：

| 维度 | 指标 | 绿灯 | 黄灯 | 红灯 |
|:-----|:-----|:----:|:----:|:----:|
| 技术质量 | 测试通过率 | ≥ 95% | 90-95% | < 90% |
| 技术质量 | 缺陷逃逸率 | ≤ 5% | 5-15% | > 15% |
| 业务质量 | 功能使用率 | 周环比增长 | 持平 | 连续两周下降 |
| 体验质量 | 核心页面加载 P75 | ≤ 2s | 2-4s | > 4s |
| 体验质量 | 用户报错率 | ≤ 0.1% | 0.1-0.5% | > 0.5% |
| 审查效率 | 审查周转时间 | ≤ 4h | 4-12h | > 12h |
| 治理效果 | 违规率 | 周环比下降 | 持平 | 周环比上升 |

**实施节奏**：
- 第一阶段（0-30 天）：只采集 10 个核心指标，做基线摸底
- 第二阶段（30-60 天）：加入红绿灯机制，开始预警
- 第三阶段（60 天后）：度量结果与调度优先级挂钩

### 4.5 季度治理圆桌

每季度（首次为 2026-10-01）召开全员治理圆桌：
- 全员可对总纲提出修改建议
- 协议执行官汇报监督运行报告
- 增长分析师汇报治理效果度量数据
- 项目总管和协议执行官联合审计执行效果
- 输出治理运行报告并更新总纲

### 4.6 长期愿景

- **监督的自我消亡**：当规则足够清晰、执行足够自动化、文化足够内化，协议执行官从「警察」变为「审计师」
- **从调度到自调度**：到 2027 年 Q1，至少 50% 的团队实现自调度——团队内部分配任务、跨组协作自行协商，总管的角色从指挥官变为资源保障者
- **治理的终点**：治理的目的是为了最终不需要治理
