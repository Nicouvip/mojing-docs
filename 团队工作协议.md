# 墨境团队工作协议

> 版本：1.0 · 生效日期：2025-07-15
> 所有角色必须遵守本协议，任务指令中不再重复说明。

---

## 一、输出规范

每个角色完成任务后，**必须**将成果写入：
```
D:\建网站\mojing-docs\output\角色名-任务简称.md
```
文件名不加日期（方便缓存命中），日期写在内容第一行。

output 文件必须包含以下字段：
- **改了什么**：列出修改的文件和改动内容
- **为什么改**：改动的理由和背景
- **对外影响**：本任务是否影响其他角色的工作？具体是什么？
- **额外发现**：任务中发现的职责范围外但需要关注的问题（可选）
- **替代建议**：如果有更好的方案，提出并附理由（可选）
- **验收命令**：用于验证改动是否正确的命令（如 tsc --noEmit）

示例：
```
D:\建网站\mojing-docs\output\后端-Turbopack修复.md
```

---

## 二、文件即通信

- **代码文件**是共享的——所有人操作同一个 `D:\建网站\mojing-app`
- **output 文件**是信息渠道——通过它了解其他角色的工作内容
- **总管（路线规划师）**是协调中枢——跨角色协调由总管负责

---

## 三、单线程修改

同一时间只允许一个角色修改同一个文件。
总管分配任务时会标注"涉及文件"，如果有冲突，必须错开顺序。

---

## 四、职责边界

任务指令中明确标注"允许修改的文件"和"不允许修改的文件"。
超出范围的改动必须先报告总管，获得许可后才能执行。

---

## 五、验收关卡

每个角色的任务完成后，总管必须验收：
1. 读取 output 文件
2. 读取实际代码文件，对比改动
3. 运行 `tsc --noEmit` 确认编译通过
4. 涉及功能的，安排 QA 验证

验收不通过 → 打回重做
验收通过 → 分配下一个任务

---

## 六、上下文保护

- 长会话（超过 15 轮）后，关键成果必须写入 output 文件
- 如果角色的会话溢出或报错，新开会话后读取 output 文件恢复上下文

---

## 七、跨角色通知

- 涉及接口变更的任务，output 文件必须写清楚"对外接口变化"
- 总管读取后，主动通知受影响的角色

---

## 八、决策日志

重要决策记录在：
```
D:\建网站\mojing-docs\output\决策日志.md
```
所有角色都能读取，确保决策透明、可追溯。

---

## 九、Git 版本控制

- 项目必须初始化 Git 仓库
- 每个角色在执行重大改动前，先做一次 git commit
- 提交信息写清"改了什么"
- 改坏时可 `git revert` 回退

---

## 十、备份铁律

- 代码推远程仓库（GitHub/Gitee）
- 用户数据迁服务端存储（P2 阶段）
- 每次重大改动前手动备份
- 你确认一个功能模块没问题后，必须立即备份

---

## 十一、环境检查

- 任务开始前，验证开发环境是否正常
- 环境问题（服务器挂了、端口被占）最高优先级修复

---

## 十二、替代建议权

角色认为有比任务指令更好的方案时，可以提出"替代建议"并附理由。
总管判断是否采纳。

---

## 十三、截止条件

任务指令中写明"失败时停止条件"，避免无限尝试。
超出预估时间 2 倍未解决 → 总管主动介入。

---

## 十四、需求确认

- 复杂任务（影响架构或超过 50 行改动）：先出方案 → 确认 → 再执行
- 简单任务（50 行以内、可回退）：直接执行

---

## 十五、外部依赖降级

- API 不可用时显示友好提示，不白屏
- 依赖锁定（pnpm-lock.yaml 必须提交）
- 核心依赖准备备选方案

---

## 十六、版本冻结

当前版本（V1 MVP）功能范围已定。
新想法记入"V2 待考虑清单"，不插入当前迭代。

---

## 十七、知识文档化

- 关键设计决策必须写入文件，不能只存在于会话中
- output 文件不只写"做了什么"，还要写"为什么这样做"
- 理由比结论更重要

---

## 十八、灰度验证

- 本地验证 → 生产验证两步走
- 一次只改一个核心模块，确认后再改下一个
- 涉及 API 变更：后端先改 → 编译验证 → 前端再改调用 → QA 端到端验证

---

## 十九、技能包使用规范

### 基本原则
1. 每个角色必须严格按照"固定使用清单"调用对应的技能包，不得跳过
2. 每个任务开始前先判断是否触发某个技能包
3. 多步骤任务：开始前 `writing-plans` → 执行中 `executing-plans` → 完成后 `requesting-code-review`

### 强制使用清单

| 场景 | 必须使用的技能包 | 违规后果 |
|:-----|:----------------|:---------|
| 写产出文件 | `mojing-output-protocol` | 产出不通过 |
| 宣称完成 | `verification-before-completion` | 打回重做 |
| 修 Bug | `systematic-debugging` | 打回重做 |
| 做 UI 组件 | `brainstorming` | 需求不通过 |
| 做代码审查 | `flow-review` | 审查无效 |
| 合并代码 | `requesting-code-review` + `finishing-a-development-branch` | 禁止合并 |
| 做 Sprint 规划 | `writing-plans` + `brainstorming` | 规划不通过 |
| 并行测试 | `dispatching-parallel-agents` | 效率不达标 |

### 各角色固定使用清单
详情见 `D:\建网站\mojing-docs\output\产品经理-技能包审计.md`

## 二十、Bug 管理制度

### 唯一事实来源

`D:\建网站\mojing-docs\Bug主清单.md` 是 Bug 管理的唯一事实来源。所有 Bug 以该清单的实时状态为准，禁止在会话/文档中另建 Bug 表。

### 各角色职责

| 角色 | 职责 | 触发时机 |
|:-----|:-----|:---------|
| **QA 测试工程师** | 每轮测试结束后更新 Bug 清单：新增 Bug（含 ID/描述/严重度/复现步骤）、已修复 Bug 标记为 ✅ | 每轮测试报告输出时 |
| **架构审查员** | 每轮审查必须检查 Bug 修复状态，确认已修复的 Bug 是否真的修好了（不只看标记） | 每轮架构审查时 |
| **总管（路线规划师）** | 每次分配任务前，先扫 Bug 清单：高优先未修复 Bug 优先排入 Sprint，已修复的移入归档并确认 QA/架构都已签字 | 每次任务分配前 |

### Bug 生命周期

```
发现 → 登记到 Bug主清单 （QA负责）
  │
  ▼
总管分配任务 → 开发修复
  │
  ▼
QA 验证 → 验证通过？
  ├── 否 → 打回，补充复现信息
  └── 是 →
       │
       ▼
架构审查员确认修复质量 →
       │
       ▼
标记为 ✅ （总管在 Bug主清单中操作）
  │
  ▼
不移除，仅归档到"已修复"区段
```

### Bug 状态定义

| 状态 | 标记 | 含义 |
|:-----|:----:|:-----|
| 未修复 | ❌ | 已登记但未分配 |
| 修复中 | 🔄 | 已派给对应角色 |
| 待验证 | 🔍 | 开发声称已修复，等待 QA 验证 |
| 已验证 | ✅ | QA 验证 + 架构确认，修复完成 |
| 过时 | ⏭ | 旧版遗留，无需修复 |

### 严重度定义

| 级别 | 标签 | 定义 | 响应时间 |
|:-----|:----:|:-----|:--------:|
| P0 | 🔴 | 功能完全不可用 / 数据丢失 / 安全风险 | 立即 |
| P1 | 🟡 | 功能部分失效 / 体验严重受损 | 当前 Sprint |
| P2 | 🟢 | 体验问题 / 非核心功能异常 | 下个 Sprint 或 backlog |
| P3 | ⚪ | 视觉/文案/优化类 | backlog |

## 十九、Bug 管理制度（全员必须遵守）

### Bug 主清单
地址：`D:\建网站\mojing-docs\Bug主清单.md` — 唯一权威 Bug 记录

### 各角色职责

| 角色 | Bug 相关职责 |
|------|------------|
| **QA 测试工程师** | 每轮测试后必须更新 Bug 清单（新增/修复/复验）。不更新 = 测试未完成 |
| **架构审查员** | 每次代码审查必须对照 Bug 清单，标注哪些已修复、哪些该修没修 |
| **总管（路线规划师）** | 每次分配任务前必须先扫 Bug 清单，优先派 P0/P1 |
| **前端/后端** | 收到任务后先读 Bug 清单，确认自己负责的 Bug 是否在列 |
| **UI/UX/产品/提示词** | 发现新问题立即报告总管，由总管决定是否入库 |

### 修复流程（不可跳步）
1. QA 或任何人发现 Bug → 报告总管
2. 总管评估严重度（P0-P3）→ 入库 Bug主清单.md
3. 总管分配修复任务 → 指定角色
4. 角色修复 → 写产出文件
5. QA 验证修复 → 更新 Bug 清单标记 ✅
6. 架构审查 → 确认修复无回归

### 铁律
- Bug 清单是所有角色的共同信息源
- 任何人发现清单与实际不符，立即报告总管
- 修复过的 Bug 不移除，只标记 ✅ 归档
- 每轮 Sprint 结束前，QA 必须出 Bug 状态总结

## 二十、角色边界防护（全员必须遵守）

### 铁律
1. 只读自己的 `tasks/角色名.md` 任务文件，**禁止读别人的任务文件**
2. 只读本任务指定的参考资料，**禁止读 output/ 目录下其他角色的报告**
3. 不做超出角色职责的事——角色定义见 `全员角色设定-强化版.md`
4. 发现自己在做不属于自己的事 → **立即停止** → 回复"越界了，请总管确认"
5. 只出报告/方案/代码 → **禁止替其他角色分配任务**
6. **禁止读取不是自己写的输出文件**，避免被误认为是自己写的
7. **主动防御**：禁止在未接到总管指令时主动浏览 output/ 目录、读取其他角色会话、分析全局项目状态

### 违规记录
- 前端技术负责人曾以总管身份分配任务 → 已制止
- UI设计师曾读取架构审查员报告 → 已制止
