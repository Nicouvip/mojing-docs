# 需求文档：回收站自动清理定时器

## 元信息
- **需求编号：** REQ-A3
- **优先级：** P0 (Must)
- **RICE 评分：** 24/100
- **涉及角色：** 前端技术负责人
- **预估工作量：** 0.5 人天
- **依赖：** 无

---

## 用户故事

> **As a** 使用墨境写作的作者
> **I want** 回收站中的章节在 30 天后自动彻底删除
> **So that** 我的 localStorage 不会因为堆积的已删除章节而膨胀，且删除逻辑与界面提示一致

**现状问题：** 回收站弹窗显示"30天后自动清除"的文字提示，但代码中没有任何定时清理或过期删除逻辑。删除的章节会永久留在回收站中，既不清理，也浪费存储空间。

---

## 功能拆解

### 3.1 在章节数据结构中添加删除时间戳
文件：`src/lib/store.ts`

当前软删除只标记 `isDeleted: true`，需要增加 `deletedAt` 字段：
```typescript
// 删除章节时
deleteChapter: (chapterId: string) => {
  setChaptersAll(prev => prev.map(ch => 
    ch.id === chapterId 
      ? { ...ch, isDeleted: true, deletedAt: Date.now() } 
      : ch
  ));
  saveClient('mojing_chapters', chaptersAll);
}
```

### 3.2 类型更新
文件：`src/lib/types.ts`

`Chapter` 类型中增加 `deletedAt?: number` 字段（可选，兼容已有数据）。

### 3.3 清理逻辑实现
在以下时机触发过期检查和清理：

**方案 A（推荐）：懒清理 + 启动时清理**
- 回收站打开时：检查所有 `isDeleted` 章节，计算 `(Date.now() - deletedAt) > 30 * 24 * 60 * 60 * 1000`，符合条件的彻底从数组中移除
- 应用启动时（首页加载 / 编辑器加载）：做一次同样的检查
- **优点：** 不浪费定时器资源，每次用户访问回收站时自动清理

**备选方案 B：setInterval 定时器**
- 应用启动后设置定时器，每 24 小时检查一次
- **缺点：** 页面不关闭一直运行，浪费资源，不推荐

### 3.4 用户提示
- 被自动清理的章节不弹窗提示（避免惊吓用户）
- 可在回收站底部加一行小字："⚠️ 今日自动清理了 X 篇过期章节"

---

## 验收标准

1. ✅ 删除章节时，`deletedAt` 时间戳被正确记录
2. ✅ 打开回收站时，过期（>30天）的章节自动从数据中移除，回收站列表不显示
3. ✅ 未过期（<30天）的章节正常显示，可恢复/彻底删除
4. ✅ 页面刷新后，清理逻辑同样生效（启动时检查）
5. ✅ 已有已删除章节的旧数据兼容（无 `deletedAt` 字段的视为立即过期，下次打开清理）

---

## 边界条件

- 旧数据过渡：项目中已有通过 `deleteChapter` 软删除的章节，这些章节没有 `deletedAt` 字段。处理方式：如果 `isDeleted` 为 true 但无 `deletedAt`，视为已过期，直接清理
- 如果用户手动"彻底删除"，则不写入 `deletedAt`，直接移出数组
- 30 天计算使用 `Date.now()` 毫秒时间戳，不依赖服务端时间

---

## 团队备注

- 这是纯前端逻辑，不涉及后端
- `deletedAt` 为可选字段（`deletedAt?: number`），不破坏现有类型
- 如果后续迁移到服务端存储，清理逻辑需要迁移到服务端定时任务
