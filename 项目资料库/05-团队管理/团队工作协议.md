# 墨境团队工作协议

> 版本：1.0 · 生效日期：2025-07-15
> 所有角色必须遵守本协议，任务指令中不再重复说明。

---

## 一、输出规范

每个角色完成任务后，**必须**将成果写入：
```
D:\建网站\mojing-docs\output\角色名-日期-任务简称.md
```

output 文件必须包含以下字段：
- **改了什么**：列出修改的文件和改动内容
- **为什么改**：改动的理由和背景
- **对外影响**：本任务是否影响其他角色的工作？具体是什么？
- **额外发现**：任务中发现的职责范围外但需要关注的问题（可选）
- **替代建议**：如果有更好的方案，提出并附理由（可选）
- **验收命令**：用于验证改动是否正确的命令（如 tsc --noEmit）

示例：
```
D:\建网站\mojing-docs\output\后端-2025-07-15-Turbopack修复.md
```

---

## 二、文件即通信

- **代码文件**是共享的——所有人操作同一个 `D:\建网站\mojing-app`
- **output 文件**是信息渠道——通过它了解其他角色的工作内容
- **总管（路线规划师）**是协调中枢——跨角色协调由总管负责

---

## 三、单线程修改

同一时间只允许一个角色修改同一个文件。
总管分配任务时会标注"涉及文件"，如果有冲突，必须错开顺序。

---

## 四、职责边界

任务指令中明确标注"允许修改的文件"和"不允许修改的文件"。
超出范围的改动必须先报告总管，获得许可后才能执行。

---

## 五、验收关卡

每个角色的任务完成后，总管必须验收：
1. 读取 output 文件
2. 读取实际代码文件，对比改动
3. 运行 `tsc --noEmit` 确认编译通过
4. 涉及功能的，安排 QA 验证

验收不通过 → 打回重做
验收通过 → 分配下一个任务

---

## 六、上下文保护

- 长会话（超过 15 轮）后，关键成果必须写入 output 文件
- 如果角色的会话溢出或报错，新开会话后读取 output 文件恢复上下文

---

## 七、跨角色通知

- 涉及接口变更的任务，output 文件必须写清楚"对外接口变化"
- 总管读取后，主动通知受影响的角色

---

## 八、决策日志

重要决策记录在：
```
D:\建网站\mojing-docs\output\决策日志.md
```
所有角色都能读取，确保决策透明、可追溯。

---

## 九、Git 版本控制

- 项目必须初始化 Git 仓库
- 每个角色在执行重大改动前，先做一次 git commit
- 提交信息写清"改了什么"
- 改坏时可 `git revert` 回退

---

## 十、备份铁律

- 代码推远程仓库（GitHub/Gitee）
- 用户数据迁服务端存储（P2 阶段）
- 每次重大改动前手动备份
- 你确认一个功能模块没问题后，必须立即备份

---

## 十一、环境检查

- 任务开始前，验证开发环境是否正常
- 环境问题（服务器挂了、端口被占）最高优先级修复

---

## 十二、替代建议权

角色认为有比任务指令更好的方案时，可以提出"替代建议"并附理由。
总管判断是否采纳。

---

## 十三、截止条件

任务指令中写明"失败时停止条件"，避免无限尝试。
超出预估时间 2 倍未解决 → 总管主动介入。

---

## 十四、需求确认

- 复杂任务（影响架构或超过 50 行改动）：先出方案 → 确认 → 再执行
- 简单任务（50 行以内、可回退）：直接执行

---

## 十五、外部依赖降级

- API 不可用时显示友好提示，不白屏
- 依赖锁定（pnpm-lock.yaml 必须提交）
- 核心依赖准备备选方案

---

## 十六、版本冻结

当前版本（V1 MVP）功能范围已定。
新想法记入"V2 待考虑清单"，不插入当前迭代。

---

## 十七、知识文档化

- 关键设计决策必须写入文件，不能只存在于会话中
- output 文件不只写"做了什么"，还要写"为什么这样做"
- 理由比结论更重要

---

## 十八、灰度验证

- 本地验证 → 生产验证两步走
- 一次只改一个核心模块，确认后再改下一个
- 涉及 API 变更：后端先改 → 编译验证 → 前端再改调用 → QA 端到端验证
